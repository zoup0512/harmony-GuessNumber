# 《数字解密》HarmonyOS 版设计文档

| 项目 | 内容 |
| --- | --- |
| 应用名称 | 数字解密（Guess Number / 1A2B） |
| Bundle Name | com.razor.game.h.guess |
| 目标平台 | HarmonyOS 手机 / 平板（Stage 模型） |
| 编译版本 | compileSdk / targetSdk / compatibleSdk = 6.0.2(22) |
| 开发语言 | ArkTS（ArkUI 声明式范式） |
| 版本 | 1.0.0 |

## 一、项目概述

本项目将一款控制台 C 语言"猜数字（1A2B）"游戏移植为 HarmonyOS 原生手机应用。原版通过 printf/scanf 完成交互，本版参考 Wordle、Sudoku 等主流益智游戏的移动端设计惯例，重构为深色沉浸式界面：瓷砖化的数字展示、定制数字键盘、可视化的机会计量、可回溯的猜测历史，并在原玩法之上补充了难度分级、限时计时、单次提示与战绩持久化等现代功能。

## 二、与原 C 版的功能对照

| 原 C 程序 | 作用 | 鸿蒙版实现 |
| --- | --- | --- |
| `CreateNum()` | 生成各位互不相同的随机数（`rand`+查重重试） | `GameEngine.generateSecret()`，Fisher–Yates 洗牌取前 n 位，天然无重复，首位可为 0（与原版一致） |
| `JudgeA()` | 统计"数字对且位置对" | `GameEngine.judge()` 中的 `a` 计数，界面上以 🥇 金牌呈现 |
| `JudgeB()` | 统计"数字对但位置错" | `GameEngine.judge()` 中的 `b` 计数，界面上以 🥈 银牌呈现 |
| `JudgeInput()` | 校验输入不重复、均为 0–9 | 交互前置化：定制键盘只提供 0–9，已输入的数字自动置灰禁用，非法输入无从产生 |
| `scanf` 整数拆位 | 读取 4 位整数并拆分 | 逐键输入，直接以字符串保存，规避原版"首位为 0 时整数丢位"的隐患 |
| 8 次机会硬编码 | 固定 4 位 / 8 次 | 难度分级：3 位·6 次 / 4 位·8 次 / 5 位·10 次 |
| 开场大段说明文字 | 规则说明 | 主页"玩法规则"半模态弹层，图标化分条呈现 |
| `system("cls")` / `getch()` | 清屏与暂停 | 页面导航（Navigation/NavDestination）与结果浮层取代 |
| 猜中 / 次数用尽提示 | 胜负结算 | 结果浮层：揭示答案瓷砖、用时与次数统计、再来一局 / 返回主页 |

## 三、玩法规则

1. 系统随机生成一组各位互不相同的数字（首位可为 0）。
2. 每次提交一组猜测，系统以奖牌反馈：🥇 金牌 = 数字对且位置对的个数，🥈 银牌 = 数字对但位置错的个数（内部对应经典 1A2B 规则的 A/B）。例：答案 5234，猜 5346 → 🥇×1 🥈×2。
3. 在限定次数内出现 nA（n 为位数）即胜利；机会用尽则失败并公布答案。
4. 每局可使用一次提示，随机揭示一位密码，代价为消耗 1 次机会（剩余机会不足 2 次时不可用）。

## 四、UI / 交互设计

### 4.1 视觉方向

整体定位为"深夜解密终端"：深靛蓝到暗紫的垂直渐变背景（#10142E → #241242），半透明白色玻璃卡片承载信息，行动色为暖金渐变（#F5B441 → #E08A2E）。反馈符号面向 12 岁以下目标用户做了具象化设计：不用抽象的 A/B 字母，而用 🥇 金牌（数字对、位置也对）与 🥈 银牌（数字对、位置错）——奖牌天然自带"金 > 银"的等级常识，学龄前儿童也能一眼理解"全对拿金牌、差一点拿银牌"；配色随之固定为金 #F5B441 / 银 #C9D2E3，红色 #F0616D 表示失败与删除。签名元素是贯穿全应用的"密码瓷砖"：主页主视觉、对局输入区、历史记录、结果揭示均使用同一套圆角瓷砖语言。

### 4.2 页面结构

主页（Index）：
- 主视觉：微倾斜的 1A2B 瓷砖 + 应用名 + 标语；
- 战绩卡：对局数 / 胜率 / 最佳次数 / 当前连胜；
- 难度选择：入门（3 位·6 次）、经典（4 位·8 次）、大师（5 位·10 次）三张卡片，选中态金色渐变高亮；
- 主行动按钮"开始破解"；
- "玩法规则"入口，以半模态面板（bindSheet）展示。

对局页（GamePage）：
- 顶栏：返回、计时胶囊（mm:ss）、剩余机会胶囊；
- 机会圆点计量条：已用次数实心金点，一眼可读进度；
- 当前输入区：与位数等量的瓷砖槽位，下一输入位以金色描边指示"光标"；
- 猜测历史：最新在上的列表，每行为序号 + 数字瓷砖 + 🥇×n（金）/ 🥈×m（银）胶囊徽标；空态给出引导文案；
- 定制键盘：3×3 数字 + 底行"⌫ / 0 / ✓ 破解"，已用数字与超长输入自动禁用，凑满位数后提交键点亮为金色渐变；
- 提示按钮：键盘上方，使用后置灰并以 Toast 告知揭示结果；
- 结果浮层：半透明遮罩 + 居中卡片，胜利显示 🎉 与用时/次数，失败显示 💥 并以瓷砖揭示正确答案，提供"再来一局 / 返回主页"。

### 4.3 交互细节

- 防错前置：重复数字在键盘层直接禁用，取代原版"输入后报错重试"；
- 自动提交：数字输满位数后延时 280ms 自动破解（延时既保证末位瓷砖完成渲染，也给误触留出删除窗口），"破解"键保留为手动兜底；
- 动效克制：难度卡选中 180ms 缓出过渡、结果浮层 260ms 淡入、主按钮金色投影，不做多余装饰动画；
- 返回处理：物理/手势返回与返回按钮均停表并出栈；从对局页返回主页时通过 pushPathByName 的 onPop 回调自动刷新战绩。

### 4.4 AI 军师（智能提示）

在"随机揭示一位"的传统提示之外，游戏内置了一个**只凭公开线索推理、从不偷看答案**的 AI 军师（每局限 2 次，每次消耗 1 次机会以维持平衡）：

- **候选集约束求解**：枚举全部各位互不相同的排列（3/4/5 位分别为 720 / 5040 / 30240 组），保留对每条历史 xAyB 线索都能复现相同反馈的候选，实时给出"剩余可能密码数"；
- **数字判定盘**：0–9 每个数字标注三态——在所有候选中都出现（必含，绿色）、都不出现（必排除，划线变暗）、不确定（中性），把玩家该做的逻辑推演可视化；
- **最优猜测推荐**：候选 ≤150 时采用 Knuth 式贪心，选取使期望剩余候选数 \(\mathbb{E} = \sum_k n_k^2 / N\) 最小的猜测（\(n_k\) 为各 xAyB 反馈桶的大小）；候选较多时从候选集中随机取一个合法猜测，支持"一键填入"直接开猜。经 200 局随机模拟验证，真实答案始终位于候选集内、判定零误报；AI 全程代打 4 位局平均 5.34 步破解。

产品考量：AI 的强力分析用"消耗机会 + 限次"定价，玩家在"省机会自己推理"与"花机会买信息"之间做策略权衡，反而加深了对游戏的理解。

### 4.5 新手引导（Onboarding）

1A2B 的规则一句话能说完，但"xAyB 反馈到底怎么算"是新手最大的理解门槛，因此引导的设计原则是 **Show, don't tell —— 先演示、再上手**，采用四步渐进式结构（TutorialPage）：

| 步骤 | 内容 | 交互形式 |
| --- | --- | --- |
| ① 目标认知 | 四个金色「?」瓷砖 + 一句话说明"系统藏了一组互不相同的数字" | 静态展示 |
| ② 读懂线索（核心） | 以 答案 5234 / 猜 5346 为例逐步高亮：点一下「5」变绿标 A（位置对），再点一下「3」「4」变蓝标 B（位置错）、「6」变暗（不存在），最终弹出 1A2B 徽标 | 点按驱动的分段动画，节奏由玩家自己掌握 |
| ③ 推理策略 | 三张策略卡：0A0B = 全排除、只有 B = 换顺序、A 变多 = 方向对 | 卡片列表 |
| ④ 上手练习 | 内嵌一局真实的 3 位迷你对局（复用 GameEngine，不限次数），每猜一次，"教练"气泡即时把 xAyB 翻译成人话（如"0A0B：1、2、5 都不在密码里，全部排除！"），猜中即出师 | 完整交互，与正式对局同一套键盘与瓷砖 |

关键设计决策：

- **颜色语义提前建立**：引导中 A 绿 / B 蓝 / 排除变暗，与正式对局、历史记录、结果页完全一致，玩家学一次终身受用；
- **练习局零压力**：不限次数、不计入战绩、可随时跳过，降低学习焦虑；
- **触发与回看**：首次启动自动进入（Preferences 中 `tutorial_done` 标记，完成或跳过均置位），主页底部保留「新手教学」入口可随时重温；
- **实现方式**：TutorialPage 注册为 Navigation 的 `tutorial` 路由，与对局页共用同一 NavPathStack；步骤切换用 @State 驱动，第②步的分段高亮用属性动画（240ms EaseOut）完成，无需额外动画库。

## 五、技术架构

### 5.1 分层

```
entry/src/main/ets
├── entryability/EntryAbility.ets   应用入口：加载首页、沉浸式窗口设置
├── pages/Index.ets                 主页 + Navigation 容器（@Entry）
├── views/GamePage.ets              对局页（NavDestination 子页）
├── views/TutorialPage.ets          新手引导页（四步渐进式教学）
├── model/GameEngine.ets            纯逻辑：生成密码、判定 A/B、提示
├── model/StatsStore.ets            战绩持久化 + 引导完成标记（preferences）
└── common/Theme.ets                视觉常量（配色、圆角）
```

- 逻辑与 UI 分离：GameEngine 不依赖任何 UI，可独立单测；
- 导航：采用官方推荐的 Navigation + NavPathStack（Stack 模式），难度参数经 `pushPathByName('game', digits, onPop)` 传递，`onReady` 中读取；
- 状态管理：页面内 @State 驱动渲染；NavPathStack 经 @Provide/@Consume 跨层共享；
- 持久化：Preferences 键值存储 played / wins / best / streak，胜负结算时异步落盘；
- 计时：setInterval 秒级计时，页面销毁与对局结束时清理，避免泄漏。

### 5.2 关键算法

密码生成采用 Fisher–Yates 洗牌：对 0–9 洗牌后取前 n 位，时间复杂度 \(O(10)\)，相比原版"随机后查重重试"无重试开销且分布均匀。判定阶段对猜测串逐位在密码中定位：命中同下标记 A，命中异下标记 B，复杂度 \(O(n^2)\)（n ≤ 5，可忽略）。由于双方均无重复数字，恒有

\[ A + B \le n \]

提示功能从"尚未被任何历史猜测在正确位置命中"的位中随机揭示一位，避免浪费在玩家已确认的信息上。AI 军师的候选集过滤、数字三态判定与贪心推荐算法详见 4.4 节，全部实现于 GameEngine.analyze() 静态方法，与 UI 完全解耦。

## 六、工程配置

- 根 `build-profile.json5`：compileSdk / compatibleSdk / targetSdk 均为 `6.0.2(22)`，runtimeOS 为 HarmonyOS；
- `AppScope/app.json5`：bundleName 为 `com.razor.game.h.guess`；
- `hvigor/hvigor-config.json5`：modelVersion `6.0.2`；
- 权限：无需任何敏感权限，纯本地离线运行。

## 七、构建与运行

1. 使用 DevEco Studio（6.0.2 及以上，含 API 22 SDK）选择 File → Open 打开工程根目录，等待 Sync 完成（首次会自动安装 hvigor 依赖；若 IDE 提示升级 hvigor/modelVersion，按提示一键同步即可）。
2. 连接 HarmonyOS 真机（或启动模拟器），在 File → Project Structure → Signing Configs 勾选 Automatically generate signature 完成自动签名。
3. 点击 Run ▶ 部署运行；也可通过 Build → Build Hap(s) 产出安装包。
